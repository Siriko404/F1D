# H9: CEO Clarity and Takeover Hazards — Provenance Documentation

## Suite Overview

### A1) Identity

| Field | Value |
|-------|-------|
| **Suite ID** | H9 |
| **Research Question** | Does CEO speech clarity affect the probability of receiving a takeover bid? Do clearer CEOs face lower takeover hazard, particularly for uninvited (hostile/unsolicited) bids? |
| **Estimation Unit** | Firm-year interval (counting-process format for survival analysis) |
| **Primary Keys** | `gvkey` (firm identifier), `start`/`stop` (counting-process time bounds) |

### A2) Model Family and Estimator

| Field | Value |
|-------|-------|
| **Model Family** | Survival Analysis — Cox Proportional Hazards with Time-Varying Covariates |
| **Estimator** | `lifelines.CoxTimeVaryingFitter` (counting-process format) |
| **Objective/Link** | Partial likelihood maximization; log-hazard ratio linear in covariates |
| **Within-Transformation** | None (PH assumption handles time-invariance) |
| **Weighting** | None (equal-weighted firm-year intervals) |

### A3) Outcome(s) / Target Definition

| Field | Value |
|-------|-------|
| **Event Variable(s)** | `Takeover` (all bids), `Takeover_Uninvited` (hostile+unsolicited), `Takeover_Friendly` (friendly+neutral) |
| **Event Coding** | Binary: 1 if firm received takeover bid in that interval, 0 otherwise (censored) |
| **Censoring Definition** | Right-censored at end of sample period (2018) or last observed call year |
| **Timing Convention** | Duration = years from first call to takeover announcement or censoring |

### A4) Main RHS and Controls

| Category | Variables |
|----------|-----------|
| **Key Independent Variables** | Varies by variant (see below) |
| **Interaction Terms** | None |
| **Control Blocks** | `FINANCIAL_CONTROLS`: Size, BM, Lev, ROA, EPS_Growth, StockRet, MarketRet, SurpDec |
| **Sample Filters** | Main sample: FF12 codes 1-7, 9-10, 12 (excludes Finance=11, Utility=8) |

**Model Variants** (extended 2026-03-10):

| Variant | Key RHS Variables | Description |
|---------|-------------------|-------------|
| **CEO** | `ClarityCEO`, `CEO_QA_Uncertainty_pct` | Original CEO clarity (4.1.1) model |
| **CEO_Residual** | `CEO_Clarity_Residual`, `CEO_QA_Uncertainty_pct` | Residualized CEO clarity after controlling for firm/year FE |
| **Manager_Residual** | `Manager_Clarity_Residual`, `Manager_QA_Uncertainty_pct` | Residualized Manager clarity after controlling for firm/year FE |

**NOTE**: The original Regime variant (ClarityManager + Manager_QA_Uncertainty_pct) was deprecated on 2026-02-28 due to missing ClarityManager scores. The Manager_Residual variant (added 2026-03-10) is semantically different — it uses the *residualized* Manager uncertainty component, not the raw ClarityManager FE.

### A5) Fixed Effects + Variance Estimator

| Field | Value |
|-------|-------|
| **Stratification** | None |
| **Frailty/Random Effects** | None |
| **Standard Errors** | Robust (default from lifelines) |
| **Cluster Type** | None at estimation level (firm is the analysis unit) |
| **Small-Sample Corrections** | None |

### A6) Model-Family Addendum (Survival / Hazards)

| Field | Value |
|-------|-------|
| **Time Origin (t0)** | Year of first observed earnings call for each firm |
| **Time Scale + Units** | Years (integer calendar years) |
| **Duration Construction** | `stop - start` per interval; cumulative from entry year |
| **Event Definition** | `Takeover=1` if SDC reports bid received; 0 otherwise |
| **Censoring Definition** | Firms without bid by 2018 or last call year |
| **Admin Censor Date** | 2018-12-31 (end of sample period) |
| **Risk Set Construction** | Counting-process (start/stop) format — one row per firm-year at risk |
| **Late Entry/Left Truncation** | Entry at first call year (left-truncated before entry) |
| **Ties Method** | Efron (default for CoxTimeVaryingFitter) |
| **Competing Risks Approach** | Cause-specific Cox models (separate models for All/Uninvited/Friendly) |
| **Cause/Event-Type Coding** | `Takeover_Type`: 'Uninvited' (Hostile+Unsolicited), 'Friendly' (Friendly+Neutral), 'Unknown' (other), 'None' (no bid) |

## G. Outliers, Missing Data, Scaling, and Model-Specific Data Rules

### Winsorization

**Strategy**: Per-year winsorization applied at engine level BEFORE panel construction.

**Variables Winsorized**:
- **CompustatEngine**: Size, BM, Lev, ROA, EPS_Growth (per-year 1%/99%)
- **CRSPEngine**: StockRet, MarketRet (per-year 1%/99%)
- **LinguisticEngine**: All `_pct` columns (per-year 0%/99% — upper bound only)

### Missing Data Policy

**Listwise Deletion**: Rows with missing values in required regression variables are dropped before estimation:
```python
# run_h9_takeover_hazards.py:387-391
needed_cols = [START_COL, STOP_COL, "gvkey", event_col] + covariates
needed_cols = [c for c in needed_cols if c in df.columns]
df_clean = df[needed_cols].dropna(subset=[START_COL, STOP_COL, event_col] + covariates).copy()
```

### Survival-Specific Rules

**Invalid Durations**: Duration (stop - start) must be ≥ 0. Intervals with negative duration are dropped.

**Left Truncation**: Firms enter the risk set at their first observed call year. No observation before entry.

**Right Censoring**: Firms without takeover event are censored at min(last_call_year, 2018).

**Competing Risks**: For cause-specific models (Uninvited, Friendly), events of other causes are treated as censored at the event time. Unknown takeover types (n=277) are treated as censored in both cause-specific models.

### Issue 2: Unknown Takeover Types Treated as Censored (INVESTIGATED 2026-03-10)

**Description**: 37 firms (277 interval-level observations) have `Takeover_Type='Unknown'` (neither Uninvited nor Friendly). These are treated as censored in both cause-specific models.

**Root Cause (Investigated 2026-03-10)**:
- SDC "Deal Attitude" values that map to "Unknown": **"No Applicable"** (81 deals in filtered SDC [US Public, 2002-2018, Completed/Withdrawn/Pending], 37 matched to sample)
- This is a legitimate SDC classification for deals where attitude doesn't apply (e.g., negotiated transactions without formal bid process)
- Cannot be reclassified without manual review of each deal

**Impact**: The cause-specific models (Uninvited, Friendly) may underestimate the true hazard because some events are misclassified as censored.

**Fix Applied**:
- Added logging to `takeover_indicator.py` to report Unknown attitude sources in future runs

**Verification Status**: INVESTIGATED. "No Applicable" deals are legitimate unknowns that cannot be reclassified algorithmically. Document as data limitation.

### Issue 3: Few Uninvited Events (N=45)

**Description**: The Cox CS Uninvited model has only 45 event firms, which may limit statistical power.

**Impact**: Wide confidence intervals; potential for overfitting.

**Recommended Fix**: Interpret Uninvited results with caution. Consider pooling Uninvited with Unknown or reporting confidence intervals prominently.

**Verification Status**: VERIFIED via model_diagnostics.csv (n_event_firms=45 for Uninvited).

**Impact**: Listwise deletion reduces the CEO variant sample from 27,773 to 16,519 firm-year intervals.

**Verification Status**: VERIFIED via hazard_ratios.csv.

### Issue 6: Negative Duration Intervals (RESOLVED 2026-03-10)

**Description**: 8 rows in the counting-process panel had `start > stop`, resulting in negative durations. This was a data construction bug in the counting-process panel build.

**Root Cause**: When a firm's takeover occurred BEFORE its first observed call (takeover_year < entry_year), the fallback row creation in `build_counting_process_panel()` produced `start = entry_year - 1` and `stop = exit_year = takeover_year`, resulting in `start > stop`.

**Fix Applied (2026-03-10)**:
1. Added validation in `build_h9_takeover_panel.py` to skip firms with `exit_year < entry_year` (lines 386-396)
2. Added assertion after panel construction to catch any remaining negative durations (lines 460-468)
3. Firms with takeover before first observed call are now excluded with warning

**Verification Status**: RESOLVED. Invalid firms are now skipped with warning; negative duration assertion ensures data integrity.
