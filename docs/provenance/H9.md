## A. Suite Overview

### A1) Identity

| Field | Value |
|-------|-------|
| Suite ID | H9 |
| Suite name | Takeover Hazard Models |
| Research question | Does clarity in speech increase the likelihood of receiving a takeover bid, especially an uninvited bid? |
| Estimation unit | Call-to-call counting-process interval (firm Ã— interval) |
| Primary keys | `gvkey` (firm) Ã— `start`/`stop` (interval endpoints in days since 2000-01-01) |

### A2) Model family and estimator

| Field | Value |
|-------|-------|
| Model family | Survival / Hazard â€” Cox Proportional Hazards (time-varying covariates) |
| Estimator | `lifelines.CoxTimeVaryingFitter` (counting-process format) |
| Objective | Partial likelihood (Cox) |
| Optimizer | `lifelines` default (Newton-Raphson via internal implementation) |
| Tie method | Efron (lifelines default for `CoxTimeVaryingFitter`) |
| Regularization | None |
| Weighting | None (no weights/offset/exposure) |
| Model 1 | Cox PH â€” All Takeovers (event = `Takeover`) |
| Model 2 | Cause-specific Cox PH â€” Uninvited (event = `Takeover_Uninvited`) |
| Model 3 | Cause-specific Cox PH â€” Friendly (event = `Takeover_Friendly`) |

### A3) Outcome / target definition

| Outcome | Definition | Coding |
|---------|-----------|--------|
| `Takeover` | Any takeover bid (Completed, Withdrawn, Pending) for US public target in sample period | 1 in the interval containing the bid announcement; 0 otherwise |
| `Takeover_Uninvited` | Bid with Deal Attitude = Hostile or Unsolicited | 1 if `Takeover=1` AND `Takeover_Type='Uninvited'`; 0 otherwise |
| `Takeover_Friendly` | Bid with Deal Attitude = Friendly or Neutral | 1 if `Takeover=1` AND `Takeover_Type='Friendly'`; 0 otherwise |
| Unknown-type events | 26 events with `Takeover_Type='Unknown'` | Correctly treated as censored in both cause-specific models |

**Timing:** Event occurs at SDC `Date Announced`. Covariates measured at the call that opens each interval (pre-event). Intervals are call-to-call, not calendar.

**Missingness â†’ drop:** Listwise deletion on all covariates + start/stop/event. Rows with NaN clarity or controls are dropped.

### A4) Main RHS and controls

**Key independent variables (3 clarity variants, run as separate models):**

| Variant | Variable | Family | Description |
|---------|----------|--------|-------------|
| CEO | `ClarityCEO` | primary_style | CEO fixed effect from H1 clarity regression |
| CEO_Residual | `CEO_Clarity_Residual` | secondary_residual | Residualized CEO uncertainty from H0.3 extended regression |
| Manager_Residual | `Manager_Clarity_Residual` | secondary_residual | Residualized manager uncertainty from H0.3 extended regression |

**Control blocks:**

| Block | Variables | Where defined |
|-------|-----------|---------------|
| Sparse (all models) | Size, BM, Lev, ROA, CashHoldings | `run_h9_takeover_hazards.py:130-136` |
| Expanded (robustness) | Sparse + SalesGrowth, Intangibility, AssetGrowth | `run_h9_takeover_hazards.py:139-143` |

**Sample filter:** Main sample = FF12 codes 1â€“7, 9â€“10, 12 (excludes Finance ff12=11 and Utility ff12=8). Applied at `run_h9_takeover_hazards.py:277`.

### A5) Fixed effects equivalent + variance estimator

| Field | Value |
|-------|-------|
| Fixed effects | None (no entity/time FE in base models) |
| Stratified baseline hazard | Year-stratified (`strata="year"`) and Industry-stratified (`strata="ff12_code"`) as separate robustness models |
| Standard errors | **Model-based (inverse Hessian)** â€” NOT robust sandwich. `CoxTimeVaryingFitter.fit()` defaults to `robust=False`; `robust=True` raises `NotImplementedError` in lifelines 0.30.0. |
| Clustering | **NOT CLUSTERED** â€” `CoxTimeVaryingFitter` does not support `cluster_col`. SEs are model-based, non-robust, and non-clustered. |

### A6) Survival / Hazards addendum

| Field | Value |
|-------|-------|
| Time origin (tâ‚€) | 2000-01-01 (arbitrary reference date) |
| Time scale | Calendar days since 2000-01-01 |
| Duration construction | `start` = call_date âˆ’ 2000-01-01 (days); `stop` = min(next_call_date, takeover_date, censor_date) âˆ’ 2000-01-01 |

## G. Outliers, Missing Data, Scaling, and Model-Specific Data Rules

### Winsorization

All Compustat-derived controls are winsorized at 1%/99% **by fiscal year** (fyearq) in the CompustatEngine. Minimum 10 observations per year-group required; years with <10 are left unwinsorized. Applied before merge to manifest.

- **Listwise deletion** within each model: `run_cox_tv()` drops rows where any covariate or start/stop/event is NaN (`run_h9_takeover_hazards.py:441-444`)

## K. Corporate Finance Thesis Referee Assessment

### K1) Bottom-line referee verdict

This suite implements a carefully-constructed counting-process survival analysis examining whether managerial clarity affects takeover hazard. The interval construction is sound, with appropriate event-time construction, administrative censoring, interval capping, and cause-specific competing risks handling. However, the uniformly null results (all clarity coefficients statistically insignificant with p > 0.48, concordance indices near 0.50), combined with critical methodological gaps â€” particularly the **use of model-based (inverse Hessian) standard errors that are neither robust nor clustered** (a serious deficiency for a multi-interval panel; `robust=True` raises `NotImplementedError` in lifelines 0.30.0, so the fix requires switching estimator class entirely to `CoxPHFitter` or implementing block bootstrap), the **unverified proportional hazards assumption**, the **absence of a power analysis for this null-result suite**, and **substantial structural missingness** in the key explanatory variable â€” place this suite below thesis standard as currently implemented. The null finding itself is potentially informative, but the inference apparatus has material gaps that prevent a thesis committee from being confident that the null is genuine rather than an artifact of methodological limitations or insufficient statistical power.

**Overall status: SALVAGEABLE WITH MAJOR REVISIONS**

The implementation currently supports:
- **Descriptive association:** YES, with caveats about sample selection and model-based (non-robust, non-clustered) SEs
- **Predictive relation:** NO â€” concordance near 0.50 across all models indicates no predictive power
- **Causal interpretation:** NO â€” no identification strategy beyond controls
- **Structural interpretation:** NO

## N. Final Readiness Statement

**Can this suite be reproduced end-to-end from code alone?**
YES. Both panel build and estimation scripts are fully runnable with clear input/output paths, timestamped outputs, and manifest documentation.

**Can a committee/referee understand exactly how the variables are built?**
YES. Variable construction is explicit, documented in builders, and traceable to raw Compustat/SDC fields through the CompustatEngine and TakeoverIndicatorBuilder.

**Are the implemented results trustworthy for descriptive inference?**
PARTIALLY. The null association between clarity and takeover hazard is consistently observed across 36 specifications. However, **model-based (non-robust, non-clustered) standard errors** (the inference problem is worse than originally documented â€” SEs are not even heteroskedasticity-robust, and `robust=True` is not available for `CoxTimeVaryingFitter`), unverified PH assumption, absence of power analysis for the null finding, and structural missingness of the key explanatory variable introduce sufficient uncertainty that a referee cannot fully trust the precision of the null. The null conclusion itself is likely robust to SE correction (cluster-robust SEs are typically larger than model-based SEs, so corrected p-values would be even higher), but the methodological credibility gap is severe.

**Are they trustworthy for causal inference?**
NO. No identification strategy beyond controls. Known omitted variables (governance, ownership). The design supports only descriptive association.

**What is the single biggest threat to credibility?**
**Model-based (inverse Hessian) standard errors â€” neither robust nor clustered.** For a counting-process panel with ~40 intervals per firm, the SEs have no heteroskedasticity correction and no clustering. `CoxTimeVaryingFitter.fit()` defaults to `robust=False`, and `robust=True` raises `NotImplementedError` in lifelines 0.30.0. The fix requires switching estimator class entirely (to `CoxPHFitter` with `entry_col`, `cluster_col`, `robust=True`) or implementing firm-level block bootstrap. While the all-null result means this likely does not change the qualitative conclusion, it undermines the methodological credibility of the suite and would be flagged immediately by any econometrics-aware reviewer.

**What is the single highest-value next fix?**
**Switch to `CoxPHFitter` with `cluster_col='gvkey'` and `robust=True`** (requires data restructuring to use `entry_col` for late-entry handling in the counting-process format), or implement firm-level block bootstrap. This is a more substantial change than simply adding a parameter â€” it requires changing the estimator class.
