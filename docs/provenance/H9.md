## 1. Suite Overview

### 1.1 Identity

| Field | Value |
|-------|-------|
| Suite ID | H9 |
| Suite name | Takeover Hazard Models |
| Research question | Does clarity in speech increase the likelihood of receiving a takeover bid, especially an uninvited bid? |
| Estimation unit | Call-to-call counting-process interval (firm x interval) |
| Primary keys | `gvkey` (firm) x `start`/`stop` (interval endpoints in days since 2000-01-01) |
| Entrypoint (econometric) | `src/f1d/econometric/run_h9_takeover_hazards.py` |
| Entrypoint (panel) | `src/f1d/variables/build_h9_takeover_panel.py` |

### 1.2 Model Family and Estimator

| Field | Value |
|-------|-------|
| Model family | Survival / Hazard -- Cox Proportional Hazards with time-varying covariates |
| Estimator | `lifelines.CoxTimeVaryingFitter` (counting-process format) |
| Objective | Partial likelihood (Cox) |
| Optimizer | lifelines default (Newton-Raphson) |
| Tie method | Efron (lifelines default for `CoxTimeVaryingFitter`) |
| Regularization | None |
| Weighting | None |

This is the only survival analysis suite in the thesis. All other suites use OLS/IV/Probit.

### 1.3 Model Definitions

| Model | Event Column | Description |
|-------|-------------|-------------|
| Model 1: Cox PH All | `Takeover` | All takeover bids |
| Model 2: Cox CS Uninvited | `Takeover_Uninvited` | Cause-specific -- Hostile + Unsolicited bids |
| Model 3: Cox CS Friendly | `Takeover_Friendly` | Cause-specific -- Friendly + Neutral bids |

### 1.4 Outcome Definitions

| Outcome | Definition | Coding |
|---------|-----------|--------|
| `Takeover` | Any takeover bid (Completed, Withdrawn, Pending) for US public target in sample period | 1 in the interval containing the bid announcement; 0 otherwise |
| `Takeover_Uninvited` | Bid with Deal Attitude = Hostile or Unsolicited | 1 if `Takeover=1` AND `Takeover_Type='Uninvited'`; 0 otherwise |
| `Takeover_Friendly` | Bid with Deal Attitude = Friendly or Neutral | 1 if `Takeover=1` AND `Takeover_Type='Friendly'`; 0 otherwise |
| Unknown-type events | Events where Deal Attitude is not Hostile/Unsolicited/Friendly/Neutral (e.g., "Not Applicable") | Correctly treated as censored in both cause-specific models (Takeover_Uninvited=0 and Takeover_Friendly=0) |

**Timing:** Event occurs at SDC `Date Announced`. Covariates are measured at the call that opens each interval (pre-event).

**Missingness handling:** Listwise deletion on all covariates + start/stop/event per model. Rows with NaN clarity or controls are dropped at `run_h9_takeover_hazards.py:441-444`.

### 1.9 Survival Construction

| Field | Value |
|-------|-------|
| Time origin (t0) | 2000-01-01 (arbitrary reference date; `build_h9_takeover_panel.py:98`) |
| Time scale | Calendar days since 2000-01-01 |
| Duration construction | `start` = call_date - 2000-01-01 (days); `stop` = min(next_call_date, takeover_date, censor_date) - 2000-01-01 |

### 5.3 SDC Merge

- SDC `cusip6` matched to manifest-derived `cusip6` via inner join
- First bid per gvkey selected (subsequent bids dropped)
- Firms not in SDC receive Takeover=0 via `fillna(0)` at `takeover_indicator.py:210`
- Deal Attitude classification: Hostile/Unsolicited -> "Uninvited"; Friendly/Neutral -> "Friendly"; everything else -> "Unknown"

### 5.4 Counting-Process Construction

At `build_h9_takeover_panel.py:256-416`:
1. Calls sorted by (gvkey, call_date)
2. Post-takeover calls removed (not at risk)
3. Next-call date computed per firm via `.groupby('gvkey')['call_date'].shift(-1)`
4. Stop date = min(next_call_date, censor_date=2018-12-31)
5. For event firms: if takeover_date falls in (call_date, stop_date], stop truncated to takeover_date
6. Event indicator = 1 only in that interval
7. Intervals exceeding 4-year cap (1,461 days): event intervals censored (Takeover -> 0), all capped
8. Zero/negative duration intervals removed
9. Multi-event validation (ValueError if >1 event per firm)

## 6. Variable Dictionary

### 6.1 Survival Variables

#### 6.1.1 `start` -- Interval Start Time

| Field | Value |
|-------|-------|
| Definition | Call date expressed as integer days since 2000-01-01 |
| Type | Integer (days) |
| Code | `build_h9_takeover_panel.py:339` |

**Construction chain:**

1. `start_date` is loaded from the master manifest (`master_sample_manifest.parquet`) as a datetime column representing the earnings call date.
2. Converted to `call_date` via `pd.to_datetime(start_date).dt.normalize()` at `build_h9_takeover_panel.py:286`.
3. Converted to integer days: `start = (call_date - pd.Timestamp("2000-01-01")).dt.days` at line 339.

The reference date `REFERENCE_DATE = pd.Timestamp("2000-01-01")` is defined at `build_h9_takeover_panel.py:98`. This is an arbitrary origin chosen so that all start/stop values are positive integers (the earliest calls in the sample are circa 2002). The choice of reference date does not affect Cox PH estimation because the model conditions on the risk set at each event time.

#### 6.1.2 `stop` -- Interval Stop Time

| Field | Value |
|-------|-------|
| Definition | Interval close date expressed as integer days since 2000-01-01 |
| Type | Integer (days) |
| Code | `build_h9_takeover_panel.py:340` |

**Construction chain (counting-process logic at lines 256-416):**

1. Calls are sorted by `(gvkey, call_date)` at line 288.
2. Post-takeover calls are removed: any call on or after `Takeover_Date` is dropped (lines 304-312) because the firm is no longer at risk.
3. Next-call date is computed per firm: `next_call_date = groupby('gvkey')['call_date'].shift(-1)` at line 320.
4. Initial stop date: `stop_date = min(next_call_date, censor_date)` where `censor_date = pd.Timestamp("{year_end}-12-31")` (line 283). If no next call exists, `stop_date = censor_date` (line 323). If `stop_date > censor_date`, it is clamped to `censor_date` (line 324).
5. For event firms: if `Takeover_Date` falls in the interval `(call_date, stop_date]`, then `stop_date` is truncated to `Takeover_Date` (lines 327-333).
6. Converted to integer days: `stop = (stop_date - REFERENCE_DATE).dt.days` at line 340.
7. 4-year cap applied: if `stop - start > 1461` (MAX_INTERVAL_DAYS), `stop` is clamped to `start + 1461` at line 359. Event intervals exceeding the cap are censored (`Takeover` set to 0, `Takeover_Type` set to `"None"`) at lines 354-356.

#### 6.1.3 `duration` -- Interval Length

| Field | Value |
|-------|-------|
| Definition | `stop - start` in days |
| Type | Integer (days) |
| Code | `build_h9_takeover_panel.py:370` |

Used for summary statistics only; not passed to `CoxTimeVaryingFitter` (which uses `start`/`stop` directly). Intervals with `duration <= 0` are removed at lines 373-377.

#### 6.1.4 `Takeover` -- All-Bid Event Indicator

| Field | Value |
|-------|-------|
| Definition | 1 if any takeover bid announcement falls within this interval; 0 otherwise |
| Type | Binary integer |
| Code | `build_h9_takeover_panel.py:336` (panel); `run_h9_takeover_hazards.py:200` (runner uses as `EVENT_ALL_COL`) |

**Full construction chain:**

1. **SDC M&A raw data** is loaded from `inputs/SDC/sdc-ma-merged.parquet` by `TakeoverIndicatorBuilder` (`takeover_indicator.py:108`). Columns read: `Target 6-digit CUSIP`, `Date Announced`, `Deal Status`, `Deal Attitude`, `Target Public Status`, `Target Nation`.
2. **Filtering** (lines 122-129): US-only (`Target Nation == "United States"`), public targets (`Target Public Status == "Public"`), actionable statuses (`Deal Status in {"Completed", "Withdrawn", "Pending"}`), sample period (`year_start <= year(Date Announced) <= year_end`). Raw 142,457 rows filter to approximately 5,820 deals.
3. **CUSIP6 linkage** (lines 74-89): The manifest provides `cusip` (9-character ISIN-style). The first 6 characters (`cusip6`) are extracted. A unique `gvkey -> cusip6` map is built from the manifest. SDC's `Target 6-digit CUSIP` is joined to this map via inner join on `cusip6` (lines 158-162). Approximately 808 SDC deals match manifest gvkeys, covering 690 unique targeted firms.
4. **First-bid-only** (lines 166-178): For each `gvkey`, deals are sorted by `Date Announced` ascending, and `.groupby('gvkey').first()` selects only the earliest bid. This ensures at most one event per firm, as required by the counting-process construction's single-event assumption.
5. **Type classification** (lines 182-186): `Deal Attitude` is mapped: `{"Hostile", "Unsolicited"}` -> `"Uninvited"`; `{"Friendly", "Neutral"}` -> `"Friendly"`; everything else (e.g., `"Not Applicable"`) -> `"Unknown"`.
6. **Non-target fill** (lines 208-211): Firms not matched in SDC receive `Takeover=0`, `Takeover_Type="None"`.
7. **Merge onto call panel** (lines 294-301): Firm-level takeover data (one row per gvkey) is left-joined onto the call panel on `gvkey`. `Takeover` filled with 0 and cast to int; `Takeover_Type` filled with `"None"`.
8. **Interval-level assignment** (line 336): After the counting-process construction, `Takeover` is set to 1 only in the specific interval where `Takeover_Date` falls within `(call_date, stop_date]`. All other intervals for event firms receive `Takeover=0`. This is the critical step that converts firm-level event status to interval-level event indicator.
9. **4-year cap censoring** (lines 354-356): If an event interval exceeds 1,461 days, `Takeover` is reset to 0 (censored). The rationale is that attributing a takeover bid to a covariate observation from 4+ years earlier is not meaningful.
10. **Multi-event validation** (lines 384-390): After construction, each firm is validated to have at most 1 event row. A `ValueError` is raised on violation.

#### 6.1.5 `Takeover_Uninvited` -- Cause-Specific Uninvited Indicator

| Field | Value |
|-------|-------|
| Definition | 1 if `Takeover=1` AND `Takeover_Type="Uninvited"` (Deal Attitude: Hostile or Unsolicited); 0 otherwise |
| Type | Binary integer |
| Code | `build_h9_takeover_panel.py:362-364` (panel); `run_h9_takeover_hazards.py:287` (runner, redundant recomputation) |

Constructed twice (redundantly). The panel builder creates it at line 362 after 4-year cap censoring. The runner overwrites it at line 287 with an identical computation: `(Takeover == 1) & (Takeover_Type == "Uninvited")`. Both are logically equivalent.

In cause-specific Cox PH (Model 2), this serves as the event indicator. Firms with `Takeover=1` but `Takeover_Uninvited=0` (i.e., Friendly or Unknown type) are treated as censored -- standard competing-risks practice.

| Definition | 1 if `Takeover=1` AND `Takeover_Type="Friendly"` (Deal Attitude: Friendly or Neutral); 0 otherwise |
| Type | Binary integer |
| Code | `build_h9_takeover_panel.py:365-367` (panel); `run_h9_takeover_hazards.py:288` (runner, redundant recomputation) |

Same redundancy pattern as `Takeover_Uninvited`. In cause-specific Cox PH (Model 3), this serves as the event indicator.

#### 6.1.7 `Takeover_Type` -- Type Classification

| Field | Value |
|-------|-------|
| Definition | Categorical type of the first takeover bid for this firm |
| Values | `"Uninvited"`, `"Friendly"`, `"Unknown"`, `"None"` |
| Code | `takeover_indicator.py:182-186` (classification); `build_h9_takeover_panel.py:301` (fill) |

In the canonical Main sample: 73 Uninvited, 461 Friendly, 26 Unknown events.

### 7.5 Concordance Index

Harrell's C-index computed for time-varying models at `run_h9_takeover_hazards.py:321-393`:
1. Predict partial hazard for all observations
2. Compute mean partial hazard per subject across follow-up
3. Use last observation per subject for event time/indicator
4. Call `lifelines.utils.concordance_index(event_times, predicted_scores, event_observed)`

This is an approximate C-index for time-varying models. The approximation is weaker for subjects with many intervals and substantial covariate changes, because the mean partial hazard may not reflect the risk at the event/censoring time. The sign convention is correct: higher partial hazard = higher risk = shorter survival.

## 11. Identification & Inference Assessment

### 11.1 Critical: Model-Based Standard Errors

The suite uses `CoxTimeVaryingFitter` which only supports model-based (inverse Hessian) SEs. `robust=True` raises `NotImplementedError`. `cluster_col` is not available.

**Impact:** SEs are likely underestimated due to:
- Within-firm correlation across intervals (multiple intervals per firm)
- Potential heteroskedasticity in the baseline hazard

**Fix path:** Migrate to `CoxPHFitter` with pre-aggregated data (loses time-varying covariates) or implement block bootstrap clustered at the firm level. This is a substantial estimator change.

### 11.2 High: No PH Assumption Test

No Schoenfeld residual test or `check_assumptions()` call exists anywhere in the H9 codebase. The proportional hazards assumption is unverified. If the hazard ratio for clarity varies over time, the Cox model produces a weighted average of time-varying effects, which may not be interpretable.

### 11.3 High: No Governance/Ownership Controls

Standard takeover prediction models include antitakeover provisions (E-index, G-index), institutional ownership, and board characteristics. Their omission creates potential omitted variable bias.

### 11.4 Medium: Calendar Time Scale

The time scale is calendar days since 2000-01-01. Alternative scales (firm age, time since IPO, time since first call) are not tested. The choice of time scale affects the baseline hazard and can influence covariate effect estimates.

### 11.5 Medium: merge_asof Without Tolerance

Compustat financial controls are matched to calls via `merge_asof` backward with no tolerance. Stale matches (e.g., matching a 2018 call to a 2015 Compustat observation) are guaranteed to occur in the sample: any firm whose last Compustat observation is years before their last earnings call will get a stale match. The extent and impact of stale matches are undocumented.

### 11.6 Medium: Power Analysis Absent

For a null-result suite, power analysis is essential to distinguish "no effect" from "insufficient power to detect an effect." This is especially acute given the reduced event counts in residual-variant models.

### 11.7 Low-Medium: Unknown-Type Events as Censored

26 events with unknown type are treated as censored in both cause-specific models. This is standard competing-risks practice but inflates the censoring fraction and could bias cause-specific hazard estimates if the unknown events are systematically different from the known types.

## 14. Design Decisions & Rationale

### 14.1 Why CoxTimeVaryingFitter (not CoxPHFitter)?

The panel is structured as call-to-call intervals where covariates (clarity residuals, financial controls) change between calls. `CoxTimeVaryingFitter` natively handles the counting-process (start/stop) format with time-varying covariates and late entry. `CoxPHFitter` would require pre-aggregating covariates to firm-level, losing the time-varying structure.

**Trade-off:** `CoxTimeVaryingFitter` does not support robust SEs or clustering (see C-1). This is the fundamental tension in the estimator choice.

### 14.2 Why Cause-Specific Cox PH (not Fine-Gray)?

The suite uses cause-specific Cox PH for competing risks (Uninvited vs. Friendly takeovers), treating non-focal events as censored. This is standard in the M&A literature and estimates the cause-specific hazard rate. Fine-Gray would estimate subdistribution hazards, which answer a different question (cumulative incidence). Cause-specific is appropriate for etiological questions (does clarity cause takeover?).

### 14.3 Why First-Bid-Only?

SDC data is filtered to the first bid per firm (`takeover_indicator.py:167-168`). This avoids double-counting firms that received multiple bids and ensures at most one event per firm, which is required for the counting-process construction's single-event assumption.

### 14.4 Why 4-Year Interval Cap?

Intervals exceeding 1,461 days (4 years) are capped at `build_h9_takeover_panel.py:343`. This prevents extreme intervals (e.g., a firm's last call in 2005 censored at 2018-12-31, yielding a 13-year interval) from distorting the baseline hazard. Event intervals exceeding the cap are censored (Takeover -> 0) to avoid attributing takeover risk to a stale covariate observation.

### 14.5 Why Three Clarity Variants?

`ClarityCEO` is the primary construct: a CEO fixed effect estimated from the H1 clarity regression. It is time-invariant per CEO.

`CEO_Clarity_Residual` and `Manager_Clarity_Residual` are secondary constructs: residualized uncertainty from the H0.3 extended regression. They are time-varying (per call) and capture the idiosyncratic component of uncertainty not explained by firm and linguistic controls.

The three variants are estimated on structurally different sub-samples due to coverage differences. Cross-variant comparisons should be interpreted as sensitivity across clarity constructs and estimation samples, not as robustness checks on a single construct.

### 14.6 Why Exclude Finance and Utility?

Standard practice in corporate finance: financial firms (FF12=11) have fundamentally different capital structures and regulatory environments; utility firms (FF12=8) face regulated takeover barriers. Both are excluded from the Main sample to improve comparability.

### 14.7 Why Efron Tie Method?

Efron approximation is the lifelines default for `CoxTimeVaryingFitter`. It provides a better approximation to the exact partial likelihood than Breslow when ties are present, at minimal computational cost. Given that events occur at daily granularity, ties are relatively rare.
