### E2) Compustat matching (within individual builders)

Size, Lev, TobinsQ, earnings_volatility all use `CompustatEngine.match_to_manifest()`:
- Backward `merge_asof` on `gvkey` + `start_date <= datadate`
- Compustat data pre-deduplicated on `(gvkey, datadate)`, keeping last (most recent restatement)
- No explicit tolerance — matches the most recent Compustat observation

**REFEREE CONCERN:** No tolerance on the Compustat merge_asof means a call in 2018 could match a 2010 Compustat record if no intervening data exists. This is unlikely for large firms but not guarded against.

### E3) IBES matching (dispersion, lagged_dispersion, earnings_surprise_ratio)

- `dispersion`: IbesEngine computes consensus per (gvkey, fpedats), then backward merge_asof to manifest by gvkey on statpers <= start_date (tolerance 365 days).
- `lagged_dispersion`: Uses shift(1) within gvkey on consensus periods. This is a row-based lag, not a calendar-based lag, so the lag period depends on consensus frequency.
- `earnings_surprise_ratio`: Backward merge_asof by gvkey on fpedats <= start_date (tolerance 120 days).

**Note on dispersion ddof convention:** Both `IbesEngine` (`_ibes_engine.py` line 208: `"std"` in agg_dict) and `IbesDetailEngine` (`_ibes_detail_engine.py` line 300: `values.std()`) use pandas default `ddof=1` (sample standard deviation). This systematically inflates dispersion values relative to the IBES convention, especially for low-analyst-count consensus periods. Does not affect cross-sectional ranking but affects magnitude interpretation.

## G. Outliers, Missing Data, Scaling, and Model-Specific Data Rules

### Winsorization

| Variable(s) | Method | Thresholds | Applied where | Code reference |
|-------------|--------|------------|---------------|----------------|
| All linguistic `_pct` columns | Per-year | 0% / 99% (upper-only) | LinguisticEngine load | `_linguistic_engine.py` line 255 |
| Size, Lev, TobinsQ, ROA, etc. | Per-year | 1% / 99% | CompustatEngine `_compute_and_winsorize()` | `_compustat_engine.py` |
| dispersion, earnings_surprise_ratio | Pooled | 1% / 99% | IbesEngine `_build_ibes_panel()` | `_ibes_engine.py` lines 111-114 |
| loss_dummy | None | N/A | Binary variable | N/A |
| earnings_volatility | Per-year | 1% / 99% | CompustatEngine | `_compustat_engine.py` |
| delta_dispersion (new DV) | **NONE** | N/A | Not winsorized in DeltaDispersionBuilder | `delta_dispersion.py` — **REFEREE CONCERN: no winsorization on DV** |

**REFEREE CONCERN — Asymmetric winsorization on linguistic variables:** The linguistic engine uses 0%/99% (upper-only) winsorization, not symmetric 1%/99%. This means extreme low values (including zeros from calls where the speaker segment is empty or has no uncertainty words) are preserved. This is defensible for percentage variables bounded at zero but should be disclosed.

### Missing data policy

- **Listwise deletion** at regression stage: `df_reg.replace([np.inf, -np.inf], np.nan).dropna(subset=required)` (run_h5_dispersion.py line 172).
- No imputation anywhere in the pipeline.
- NaN propagation through left joins: variables that fail to match get NaN.
- **No explicit disclosure** of which variables drive the most attrition.

### Denominator protections

- `Size`: `np.where(atq > 0, np.log(atq), np.nan)` — zero/negative atq -> NaN.
- `Lev`: dlcq/dlttq fillna(0) before division; no explicit atq > 0 guard (atq = 0 -> inf -> NaN via winsorization).
- `TobinsQ`: Explicit `atq > 0` check; NaN otherwise.
- `dispersion`: `meanest == 0` -> None (filtered by `meanest_min = 0.05`).
- `earnings_surprise_ratio`: Division by |MEANEST|; filtered by `meanest_min = 0.05` in IbesEngine.

### Inf handling

- `run_h5_dispersion.py` line 172: `df_reg.replace([np.inf, -np.inf], np.nan)` before dropna.
- CompustatEngine: inf replaced with NaN after ratio computations (per MINOR-9 fix).

- **Causal verbs NOT justified:** The implementation is a within-firm panel regression of analyst dispersion level on contemporaneous uncertainty language. This supports conditional correlation only. The `delta_dispersion` redesign would move toward a short-window event-study interpretation, but it is not implemented.

## N. Final Readiness Statement

**Can this suite be reproduced end-to-end from code alone?**
NO. The current code will crash on the current panel artifact. The produced results were generated by an older, no-longer-extant version of the code. The manifest's git commit reference points to yet another version (one that used `dispersion_lead`, not `dispersion`). There is no code version in the repository that can be identified as the one that produced the latest results. This is the single most critical deficiency.

**Can a committee/referee understand exactly how the variables are built?**
PARTIALLY. The variable construction is well-traced through the builder pattern, but the dependent variable has three competing definitions across the output history (level vs. lead vs. change) with no documentation of which one is canonical or why each transition occurred.

**Are the implemented results trustworthy for descriptive inference?**
CONDITIONALLY. If the code/output mismatch is resolved, the DV is confirmed, and the PanelOLS duplicate-index behavior is verified, the within-firm panel regression with firm + year FE provides a valid conditional correlation. However, the economic magnitudes are negligible, the extreme call counts (max 38/year) suggest data contamination, and the absence of any robustness testing means even the descriptive finding is fragile.

**Are they trustworthy for causal inference?**
NO. The contemporaneous design (level of dispersion regressed on same-call uncertainty) cannot distinguish cause from effect. The `delta_dispersion` redesign would improve identification but is not implemented.

**What is the single biggest threat to credibility?**
The broken manifest provenance chain. The manifest references commit `37d34e8`, which used `dispersion_lead` as DV, while the produced outputs use `dispersion`. This is not merely a stale reference — it actively points to the wrong code. Combined with the code/output mismatch and the three-time DV evolution, a referee would conclude the provenance is unreliable.

**What is the single highest-value next fix?**
Reconcile the code, panel, and manifest: (1) identify the actual producing commit via git bisect, (2) either revert to that commit to confirm reproducibility or implement `delta_dispersion` end-to-end, (3) regenerate the manifest with the correct commit reference and clean timestamps. This simultaneously fixes the reproducibility crisis, the provenance break, and (if delta_dispersion is chosen) materially improves identification.
