### E2. CRSP-to-Manifest Linkage (BidAskSpreadChangeBuilder, StockPriceBuilder, TurnoverBuilder)

Each CRSP-dependent builder independently constructs a permno_map:
- Loads CCM linktable
- **Does NOT filter on LINKPRIM** — all CCM links (including secondary J links) are used for PERMNO mapping. Only `IbesEngine` and `IbesDetailEngine` filter on LINKPRIM in ['P', 'C']. (Verified: `grep -r "LINKPRIM" src/f1d/shared/variables/bidask_spread_change.py` returns 0 hits.)
- Date-bounded: `linkdt ≤ start_date ≤ linkenddt`
- Dedup: multiple valid links → keep most recent linkdt
- Result merged to manifest on file_name → LEFT join → row count preserved

**VERIFIED IMPLEMENTATION ISSUE:** The permno mapping is constructed independently in at least 3 builders (`BidAskSpreadChangeBuilder._build_permno_map`, `StockPriceBuilder._build_permno_map`, `TurnoverBuilder._build_permno_map`). Each contains ~70 lines of identical logic. Not a correctness issue, but a maintenance risk. The absence of LINKPRIM filtering means secondary/J links could map a gvkey to a non-primary PERMNO, potentially introducing noise or mismatches in CRSP variables. The 98.9% match rate suggests the practical impact is likely small, but the linkage is technically less restrictive than standard practice.

### E3. IBES-to-Manifest Linkage (EarningsSurpriseBuilder → SurpDec → AbsSurpDec)

- `merge_asof` backward: for each call (sorted by start_date), find most recent IBES consensus whose fiscal period ended before the call (fpedats ≤ start_date), by gvkey
- Tolerance: 120 days
- No post-filter: matching directly on fpedats eliminates need for the old ±90d FPEDATS proximity filter
- Coverage: **70.0% full panel** (79,093/112,968); **69.8% Main sample** (61,544/88,205)
- AbsSurpDec = |SurpDec|, computed in `build_h14_bidask_spread_panel.py` after SurpDec merge

**IBES engine fix (2026-03-13):** Previous implementation matched on `statpers` (latest estimate date) with 180d tolerance and an additional FPEDATS ±90d proximity filter. With IBES Detail data, `statpers = max(actdats)` and analysts routinely submit estimates after the fiscal period ends, so the old `days_to_target >= 0` filter incorrectly dropped ~62.7% of valid consensus rows. The fix switched to matching on `fpedats` (fiscal period end date) directly, removed the days_to_target filter, and tightened tolerance from 180d to 120d. Coverage improved from 21.8% to 70.0% (full panel). The old AbsSurprise (= |ACTUAL−MEANEST|/|MEANEST|) was replaced with AbsSurpDec (= |SurpDec|) to match the Dzielinski et al. surprise decile methodology.

## G. Outliers, Missing Data, Scaling, and Model-Specific Data Rules

### G1. Winsorization

| Layer | Variables | Thresholds | Pooled/By-Year | When Applied | Code Reference |
|-------|-----------|-----------|----------------|--------------|----------------|
| LinguisticEngine | All `_pct` columns | 1%/99% | Per-year | At engine load time | `_linguistic_engine.py` |
| CRSPEngine | Volatility, StockRet, MarketRet, amihud_illiq | 1%/99% | Per-year | At engine load time | `_crsp_engine.py` lines 419–451 via `winsorize_by_year()` |
| IbesEngine | dispersion, earnings_surprise_ratio | 1%/99% | Pooled (global) | At engine consensus build | `_ibes_engine.py` lines 111–114 |
| Stage 4 (L4) | delta_spread, Turnover, StockPrice, AbsSurpDec + DV variants | 1%/99% | Pooled (within regression sample) | After listwise deletion, before PanelOLS | `run_h14_bidask_spread.py:_winsorize_cols()` lines 173–179 |

**VERIFIED IMPLEMENTATION ISSUE:** PreCallSpread is NOT in `WINSORIZE_COLS` (line 185) and is NOT winsorized at Stage 4. It is also not winsorized by any engine. Given PreCallSpread has a max of 0.733 (Main sample) and a 99th percentile of 0.141, some extreme values exist. PreCallSpread is both a control variable and the pre-period component of the DV — its unwinsorized state is inconsistent with the winsorization of delta_spread.

**VERIFIED IMPLEMENTATION ISSUE:** Volatility is winsorized per-year by CRSPEngine but the comment in `run_h14_bidask_spread.py` (line 184: "Excludes Volatility — already winsorized per-year in CRSP engine") means it is winsorized on a different basis (per-year at engine level) than the other controls (pooled at regression level). This creates mixed winsorization bases.

### G2. Missing Data Policy

- **Listwise deletion:** Applied via `dropna(subset=required)` in `run_regression()` (line 222). Any observation missing any required variable is dropped.
- **Infinity handling:** `replace([np.inf, -np.inf], np.nan)` before dropna (line 222).
- **Min valid trading days:** Spread windows require ≥ `max(1, w-1)` valid spread days in each pre/post window. For ±3: min 2 days each.

### G3. Denominator Protections

- Spread formula: `2*(ASKHI-BIDLO)/(ASKHI+BIDLO)` — valid only where ASKHI > 0 AND BIDLO > 0 (mask applied at line 322–325 of bidask_spread_change.py)
- Turnover: VOL / (SHROUT × 1000) — SHROUT > 0 check (line 231–235 of turnover.py)
- EarningsSurprise (SurpDec/AbsSurpDec): |MEANEST| ≥ 0.05 required by IbesEngine (line 44); raw surprise = ACTUAL − MEANEST, then ranked into deciles

### G4. Scaling

- Size: natural log of atq (Compustat total assets)
- Volatility: annualized (×√252 × 100) — expressed in percentage points
- All uncertainty variables: expressed as percentages (word count / total words × 100, via dictionary-based computation in Stage 2)

## K. Corporate Finance Thesis Referee Assessment

### K1) Bottom-Line Referee Verdict

H14 tests whether earnings-call language uncertainty increases bid-ask spreads around the call. After the IBES engine fix and AbsSurprise→AbsSurpDec replacement (2026-03-13), the regression sample grew from ~17,500 to ~54,700 (Mgr QA spec), resolving the prior critical sample-selection issue. **The results are unambiguously null across all 6 uncertainty measures.** The one nominally significant specification — CEO Presentation Uncertainty (β₁=0.0004, p(one)=0.024) — is invalidated by two independent robustness failures: (1) the placebo coefficient is 2.8x the primary coefficient (β_placebo=0.001143 vs β_primary=0.000405), indicating the effect occurs before the call, not after it; and (2) CEO_Pres significance does not survive any alternative DV construction (closing spread, ±1 window, ±5 window are all null). ### K6) Interpretation and Claims Discipline

- **Causal language:** The one-tailed test (β > 0) implies a directional hypothesis. Given firm and year-quarter FE, the identifying variation is within-firm, across-call changes in uncertainty. This is **associative, not causal** — uncertainty language is not randomly assigned.
- **The honest interpretation is 6/6 null results.** CEO Presentation Uncertainty is nominally significant (β₁=0.0004, p(one)=0.024) on a sample of 40,986, but this finding is invalidated by: (a) the placebo coefficient being 2.8x larger than the primary coefficient, indicating pre-call confounding; (b) the finding not surviving any alternative DV construction (closing spread, ±1, ±5 all null); and (c) Bonferroni correction across 6 measures pushing p to 0.144. The other five specs are unambiguously null.
- **Economic magnitudes (for context only):** The CEO Pres β₁=0.0004 on a DV with SD ≈ 0.022. A one-SD increase in CEO Pres uncertainty (SD ≈ 0.32) would change spreads by 0.0004×0.32 = 0.000128, or 0.58% of the DV's SD. The economic magnitude is negligible.
- **P-hacking risk:** Testing 6 measures × 1 hypothesis creates a multiple testing burden. The best p-value (0.024 for CEO Pres) would not survive Bonferroni correction at 5% (0.024 × 6 = 0.144). Combined with the placebo failure and DV fragility, there is no robust significant finding.
- **Table labels accurately reflect code:** VERIFIED — the LaTeX table correctly uses one-tailed p-values and reports firm-clustered SEs. However, any presentation of the CEO_Pres result as "significant" without prominently noting the placebo failure would be misleading.
