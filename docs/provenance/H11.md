## 5. Merge & Sample Construction Audit

### 5.1 Panel builder merges (identical pattern across all three builders)

Starting frame: manifest (112,968 rows). Each variable builder output is left-joined on `file_name` with a zero-row-delta enforcement: `if after_len != before_len: raise ValueError(...)`.

| Merge step | Join key | Row delta enforcement | Notes |
|------------|----------|-----------------------|-------|
| 4 DV builders | file_name | Yes (ValueError) | Linguistic uncertainty measures |
| PRiskQ variant(s) | file_name | Yes (ValueError) | Match rates vary -- see below |
| 2 linguistic control builders | file_name | Yes (ValueError) | Analyst uncertainty, negative sentiment |
| 8 financial control builders | file_name | Yes (ValueError) | Includes BookLev (built but unused in regressions) |

Conflicting columns (other than `file_name`) are dropped before merge: `data.drop(columns=conflicting)`.

### 5.2 PRiskQ merge detail

**Base (PRiskQBuilder):**
1. Load manifest with (file_name, gvkey, start_date).
2. Compute `cal_q` from `start_date` (e.g., "2010q2").
3. Load PRisk quarterly CSV, filter to requested years, deduplicate on (gvkey, cal_q) keeping max PRisk: `sort_values("PRisk", ascending=False).drop_duplicates(subset=[...], keep="first")`.
4. Winsorize PRisk at 1/99% per year via `winsorize_by_year()`.
5. Left-join manifest to PRisk on (gvkey, cal_q).
6. Realign to manifest by `drop_duplicates(["file_name"])`.
7. Match rate: 97.6% (110,256 / 112,968).

**Lag (PRiskQLagBuilder / PRiskQLag2Builder):**
- Same pipeline but computes `cal_q_lag = _get_prev_quarter(cal_q)` (or `_get_prev2_quarter`).
- Loads PRisk data for `year_start - 1` (or additionally `year_start - 2` for lag2) to cover boundary quarters.
- Merges on (gvkey, cal_q_lag) against PRisk's (gvkey, cal_q).
- Match rates: 93.1% (lag1), 91.7% (lag2).

**Lead (PRiskQLeadBuilder / PRiskQLead2Builder):**
- Same pipeline but computes `cal_q_lead = _get_next_quarter(cal_q)` (or `_get_next2_quarter`).
- Loads PRisk data for `year_end + 1` (or additionally `year_end + 2` for lead2).
- Match rates: 95.1% (lead1), 94.8% (lead2).

## 6. Variable Dictionary

This section documents the COMPLETE construction chain for every variable in the H11 suite, from raw data source through final regression-ready form.

### 6.1 PRiskQ -- Key Independent Variable (Base Sub-suite)

| Field | Detail |
|-------|--------|
| **Panel column** | `PRiskQ` |
| **Role** | Key IV (contemporaneous) |
| **Conceptual definition** | Firm-level political risk exposure in calendar quarter Q, as measured by Hassan, Hollander, van Lent, and Tahoun (2019). PRisk captures the share of a firm's earnings call devoted to political topics, as identified by a supervised machine-learning algorithm trained on political bigrams. Higher values indicate greater political risk exposure for the firm in that quarter. |
| **External data source** | `inputs/FirmLevelRisk/firmquarter_2022q1.csv` -- distributed by Hassan et al. Data vintage: 2022q1 release. |
| **File format** | TAB-separated CSV (`sep="\t"`). Key columns: `gvkey` (string), `date` (format `"YYYYqQ"`, e.g., `"2010q2"`), `PRisk` (float). Loaded via `pd.read_csv(prisk_path, sep="\t", on_bad_lines="skip")`. |
| **Builder class** | `PRiskQBuilder` in `src/f1d/shared/variables/prisk_q.py` |

**Complete construction chain:**

1. **Load raw CSV** (`_load_prisk_quarterly`): Read tab-delimited file. 354,518 raw rows; 353,437 remain after `dropna(subset=["PRisk"])`.
2. **gvkey zero-padding**: `df["gvkey"].astype(str).str.zfill(6)` -- pads to 6 characters (e.g., `"1234"` becomes `"001234"`) to match manifest gvkey format.
3. **Parse calendar quarter**: `df["date"].apply(_parse_cal_q)` -- splits on `"q"` (case-insensitive), validates quarter in {1,2,3,4}, normalizes to lowercase format `"YYYYqQ"`. Invalid entries become `None` and are dropped.
4. **Extract year**: `df["year"] = df["cal_q"].str[:4].astype(int)`.
5. **Filter to requested years**: `df[df["year"].isin(list(years))]`.
6. **Deduplication**: `df.sort_values("PRisk", ascending=False).drop_duplicates(subset=["gvkey", "cal_q"], keep="first")` -- when multiple PRisk observations exist for the same (gvkey, cal_q), the **maximum PRisk value** is retained. This is a deliberate design choice that systematically selects the highest political risk reading.
7. **Cast to float64**: `df["PRisk"] = df["PRisk"].astype("float64")`.
8. **Per-year winsorization**: `winsorize_by_year(prisk_df, ["PRisk"], year_col="year")` from `winsorization.py`. Clips PRisk at the 1st and 99th percentiles **within each calendar year** independently. Years with fewer than 10 valid observations are not winsorized. This is NOT pooled -- each year's 1%/99% thresholds are computed from that year's distribution only. Winsorization occurs BEFORE the merge to the manifest.
9. **Compute call's calendar quarter**: From manifest, `cal_q = start_date.dt.year.astype(str) + "q" + start_date.dt.quarter.astype(str)`.
10. **Merge to manifest**: Left join on `(gvkey, cal_q)`. The manifest's gvkey is also zero-padded to 6 chars.
11. **Rename**: `PRisk` column renamed to `PRiskQ`.
12. **Post-merge dedup**: `drop_duplicates(subset=["file_name"])` -- ensures one row per call after the join (guards against one-to-many if the left join produced duplicates).
13. **Match rate**: 97.6% (110,256 / 112,968 calls matched).

### 6.2 PRiskQ_lag / PRiskQ_lag2 -- Lagged IVs (Lag Sub-suite)

| Field | `PRiskQ_lag` | `PRiskQ_lag2` |
|-------|-------------|--------------|
| **Panel column** | `PRiskQ_lag` | `PRiskQ_lag2` |
| **Role** | Key IV (1-quarter lag) | Key IV (2-quarter lag) |
| **Builder class** | `PRiskQLagBuilder` in `prisk_q_lag.py` | `PRiskQLag2Builder` in `prisk_q_lag2.py` |
| **Match rate** | 93.1% | 91.7% |

**Construction chain (identical to PRiskQ except for steps 5, 9, 10):**

Steps 1-4, 6-8 are identical to PRiskQ above.

5. **Year filter extension for boundary coverage**: Lag1 loads PRisk data for `years` PLUS `min(years) - 1` (one extra year). Lag2 loads for `years` PLUS `min(years) - 1` AND `min(years) - 2` (two extra years). This ensures that a Q1 call in the first sample year can match to PRisk from the prior year's Q4 or Q3.

9. **Compute lagged calendar quarter**:
   - `_get_prev_quarter(cal_q)` for lag1: Parses `year = int(cal_q[:4])`, `quarter = int(cal_q[-1])`. If `quarter == 1`, returns `f"{year-1}q4"`. Otherwise returns `f"{year}q{quarter-1}"`. Example: `"2010q1"` maps to `"2009q4"`.
   - `_get_prev2_quarter(cal_q)` for lag2: If `quarter == 1`, returns `f"{year-1}q3"`. If `quarter == 2`, returns `f"{year-1}q4"`. Otherwise returns `f"{year}q{quarter-2}"`. Example: `"2010q2"` maps to `"2009q4"`.
   - **No consecutive-quarter validation** is performed at this stage. The arithmetic always produces a valid quarter string. Missing data (no PRisk observation for the target firm-quarter) surfaces as NaN after the merge.

10. **Merge on shifted key**: Left join manifest on `(gvkey, cal_q_lag)` against PRisk's `(gvkey, cal_q)`. This matches call quarter Q to PRisk from quarter Q-1 (or Q-2 for lag2).

### 6.4 Dependent Variables -- Textual Uncertainty Measures (4 DVs)

All four DVs share an identical construction chain. They differ only in which speaker-section segment of the earnings call is measured.

| Panel column | Speaker | Section |
|-------------|---------|---------|
| `Manager_QA_Uncertainty_pct` | All management (CEO + non-CEO executives) | Q&A session |
| `CEO_QA_Uncertainty_pct` | CEO only | Q&A session |
| `Manager_Pres_Uncertainty_pct` | All management | Presentation section |
| `CEO_Pres_Uncertainty_pct` | CEO only | Presentation section |

**Complete construction chain:**

1. **Raw transcripts**: Earnings call transcripts from `inputs/Earnings_Calls_Transcripts/`, year-partitioned.
2. **Stage 1 -- Sample construction**: `clean_metadata.py` -> `link_entities.py` -> `build_tenure_map.py` -> `assemble_manifest.py`. Produces `master_sample_manifest.parquet` (112,968 rows). Each call is segmented into speaker turns; CEO identification uses Execucomp data.
3. **Stage 2 -- Tokenization**: `tokenize_transcripts.py` tokenizes each speaker turn. Tokenization details are upstream of Stage 3 builders.
4. **Stage 2 -- LM dictionary matching**: `build_linguistic_variables.py` counts tokens matching the Loughran-McDonald (2011) "Uncertainty" word list from `inputs/LM_dictionary/`. The uncertainty percentage is computed as: `Uncertainty_pct = (count of uncertainty words in segment) / (total word count in segment) * 100`. This is computed per call per speaker-section combination.
5. **Stage 2 output**: Year-partitioned parquet files at `outputs/2_Textual_Analysis/2.2_Variables/linguistic_variables_{year}.parquet`.
6. **LinguisticEngine loading**: The `LinguisticEngine` singleton (`_linguistic_engine.py`) loads all year-partitioned parquets, concatenates them, and applies winsorization.
7. **Winsorization at engine level**: `winsorize_by_year(combined, existing_pct_cols, year_col="year", lower=0.0, upper=0.99, min_obs=10)`. This is **upper-only per-year winsorization at the 99th percentile** (lower bound is 0%, meaning no lower clipping). Years with fewer than 10 valid observations are skipped. Computed within each calendar year independently. **CORRECTION**: The prior version of this document stated these variables use "pooled" winsorization via `_finalize_data()`. In fact, `_finalize_data()` is never called by any H11 linguistic builder; winsorization occurs inside the LinguisticEngine at load time and is per-year, upper-only.
8. **Builder extraction**: Each builder (e.g., `ManagerQAUncertaintyBuilder`) queries the engine singleton and extracts `["file_name", column]`.
9. **Merge to panel**: Left join on `file_name` with zero-row-delta enforcement.

**Note on CEO-specific DVs**: `CEO_QA_Uncertainty_pct` and `CEO_Pres_Uncertainty_pct` are NaN for calls where the CEO was not identified (approximately 32% of calls). This missingness may correlate with firm characteristics.

### 6.6 Financial Controls (7 used + 1 unused)

All financial controls are built through the `CompustatEngine` singleton (`_compustat_engine.py`). The engine loads raw Compustat quarterly data from `inputs/comp_na_daily_all/comp_na_daily_all.parquet`, computes all variables on the full panel, and caches the result. Each individual builder (e.g., `SizeBuilder`) calls `engine.match_to_manifest(manifest, root_path)`, which performs a `pd.merge_asof` matching each call's `start_date` to the nearest prior Compustat `datadate` within the same gvkey (backward direction).

**Winsorization regime for all financial controls**: The CompustatEngine applies per-year 1%/99% winsorization internally via `_winsorize_by_year(comp[col], year_col)` where `year_col` is `fyearq` (fiscal year). Years with fewer than 10 valid observations are skipped. Binary variables (`DividendPayer`) are excluded from winsorization. **CORRECTION**: The prior version of this document stated financial controls use "pooled" winsorization via `_finalize_data()`. In fact, `_finalize_data()` is never called by any H11 financial builder; winsorization occurs inside the CompustatEngine and is per-year by fiscal year.

**Compustat matching method**: `pd.merge_asof(manifest_sorted, comp_sorted, left_on="start_date", right_on="datadate", by="gvkey", direction="backward")`. This matches each call to the most recent Compustat quarterly filing on or before the call date. No maximum staleness window is enforced -- a call could theoretically match to a Compustat filing from years ago if no more recent filing exists.

### 7.1 Formula template

All regressions follow this pattern (verified in all three runners):

```
{DV} ~ 1 + {IV} + {controls} + EntityEffects + TimeEffects
```

where `{controls}` = `BASE_CONTROLS` + optional presentation control. The `1 +` intercept is included in the formula string but absorbed by `drop_absorbed=True`.

|------------|-----|-------------|

| P-13 to P-24 | PRiskQ_lead2 | 4 DVs x 3 samples |

### 7.5 Hypothesis test and p-value computation

All three runners use identical one-tailed test logic:

```python
p_one = p_two / 2 if beta_prisk > 0 else 1 - p_two / 2
h_sig = not np.isnan(p_one) and p_one < 0.05 and beta_prisk > 0
```

This tests H: beta > 0 (one-tailed). LaTeX table stars use `p_one` (one-tailed p-values). See Section 11 for the implication of applying this one-tailed test to lead (placebo) specifications.

## 11. Identification & Inference Assessment

### 11.1 Core identification strategy

The suite tests whether firm-level political risk exposure is associated with uncertainty language in earnings calls. The base variant establishes contemporaneous association. The lag variant tests whether prior-quarter political risk predicts current-quarter uncertainty language (suggestive of causation). The lead variant serves as a placebo: if future political risk is also associated with current uncertainty language, this undermines the causal interpretation by indicating either reverse causality or persistence-driven confounding.

### 11.2 Threats to identification

| Threat | Evidence | Severity |
|--------|----------|----------|
| **Lead placebo failure** | The lead runner produces significant positive coefficients for most specifications (exact counts require run output). Lead betas are reported to be substantially smaller than base betas, which is consistent with moderate PRiskQ autocorrelation driving leakage into adjacent quarters rather than equal-magnitude reverse causality. | **High** |
| **Year FE too coarse for quarterly IV** | Time FE is at the calendar YEAR level (`set_index(["gvkey", "year"])`), but PRiskQ varies at the quarterly level. Within-year aggregate shocks (e.g., macro uncertainty, market volatility) that affect both political risk and speech uncertainty are not absorbed. | **High** |
| **One-tailed test applied to lead (placebo) specs** | The lead runner docstring (lines 32-37) states `H11-Lead: beta(PRiskQ_lead) > 0` and uses `p_one = p_two / 2 if beta_prisk > 0`. For a falsification test, a two-tailed test is more appropriate: any significant coefficient (positive or negative) is informative. The one-tailed test (a) halves p-values for positive lead betas, inflating apparent significance by ~2x; (b) assigns p-values near 1.0 to negative lead betas, automatically classifying them as "passing" the placebo even if strongly significant under two-tailed testing. LaTeX table stars use these one-tailed p-values. | **Medium-High** |
| **Contemporaneous controls in lag specification** | In the lag runner, controls (Size_t, ROA_t, TobinsQ_t, etc.) are measured at time t while the IV (PRiskQ_lag) is from time t-1 or t-2. If lagged PRiskQ affects current-period financial characteristics, which in turn affect current uncertainty, then contemporaneous controls absorb part of the indirect causal channel. This is a "bad controls" concern specific to the lag specification. | **Medium** |
| **Differential estimation samples within lag/lead tables** | `prepare_regression_data` calls `dropna(subset=[..., iv_var, ...])`. Since PRiskQ_lag and PRiskQ_lag2 have different match rates (93.1% vs 91.7%), the lag1 and lag2 regressions use different subsamples. Same for lead1 vs lead2 (95.1% vs 94.8%). Comparing lag1 vs lag2 coefficients is not sample-comparable. | **Medium** |
| **Simultaneity** | PRiskQ is measured over the same calendar quarter as the earnings call. Political risk and speech uncertainty may be jointly determined. | **Medium** |
| **Endogenous CEO-DV missingness** | CEO-specific DVs have ~32% missing (not all calls have identifiable CEO turns). Missingness may correlate with firm characteristics. | **Medium** |
| **Non-unique panel index** | 96.6% of (gvkey, year) cells have > 1 call (mean 3.9, max 38). PanelOLS silently accepts; entity FE demeans at gvkey level, time FE at year level. Within-R-squared reflects within-firm-year variation, not within-firm variation. | **Medium** |
| **Presentation control as mediator** | `Manager_Pres_Uncertainty_pct` is used as a control for Q&A specs, but could be a mediator if political risk affects presentation uncertainty, which then affects Q&A. | **Low-Medium** |
| **Min-calls survivorship bias** | >= 5 calls filter drops firms with few calls, biasing toward larger/more established firms. | **Low** |

### 11.3 Standard errors and clustering

Firm-level clustering handles within-firm across-year correlation. However: (a) multiple calls per firm-year create within-cluster correlation at (gvkey, year) level not explicitly addressed -- firm clustering is likely conservative enough but untested; (b) no double-clustering (firm x time) is implemented; (c) effective cluster counts vary across specs due to different estimation samples.

## 14. Design Decisions & Rationale

| Decision | What the code does | Plausible rationale | Risk |
|----------|--------------------|---------------------|------|
| **Year FE instead of quarter FE** | `set_index(["gvkey", "year"])` for PanelOLS | Simpler; may avoid over-absorption of quarterly variation. However, PRiskQ varies quarterly, making year FE too coarse. | High: within-year confounders uncontrolled. Quarter FE is a trivially implementable fix. |
| **One-tailed test for all three sub-suites** | `p_one = p_two / 2 if beta > 0` in base, lag, and lead runners | Directional hypothesis (political risk increases uncertainty) justified for base and lag. Not justified for lead (placebo). | Medium-High for lead: inflates apparent significance of positive lead coefficients. |
| **PRiskQ dedup keeps max** | `sort_values("PRisk", ascending=False).drop_duplicates(keep="first")` | Selects the most informative observation when duplicates exist. Duplicates are likely rare. | Low: systematic upward bias possible but likely small in magnitude. |
| **Three distinct winsorization regimes** | PRiskQ: per calendar year 1%/99%; linguistic: per calendar year 0%/99% (upper-only); financial: per fiscal year 1%/99%. `_finalize_data()` is never invoked for any H11 builder. | PRiskQ has a natural year-level structure (political events cluster by year). Linguistic upper-only clipping preserves the zero floor (uncertainty cannot be negative). Financial controls use fiscal year because Compustat data is organized by fiscal year. | Low: the three regimes produce subtly different tail treatments. Linguistic variables have no lower-tail clipping, meaning extreme low-uncertainty calls pass through. The fiscal-year vs calendar-year grouping difference is minor for most firms (fiscal year = calendar year for ~70% of firms). |
| **Separate panels for base/lag/lead** | Three independent panel builders with near-identical code | Ensures each sub-suite has only the columns it needs. Avoids cross-contamination of missingness patterns. | Low for correctness, but creates ~1,800 lines of duplicated code with divergence risk. |
| **BookLev built but excluded from regressions** | Panel builders import `BookLevBuilder`; runners do not include `BookLev` in `BASE_CONTROLS` | May have been included in earlier iterations and removed from the regression spec without updating the panel builder. | Low: wasted computation only. |
| **Min-calls >= 5 applied per (sample, DV, IV)** | Computed after complete-case deletion within each combination | Ensures each firm contributes enough data for within-firm estimation. Applied per-spec rather than globally to maximize sample size. | Low: firm sets differ across specs, complicating cross-spec comparisons. |
| **Controls all contemporaneous** | Financial controls matched to the call date regardless of IV timing | Standard in the literature for base (contemporaneous) specifications. Problematic for lag specifications where it creates potential bad-controls bias. | Medium for lag spec only. |
