========================================================
0) Mission
========================================================
Your mission is to answer four questions:

(1) Is the FIRST-LAYER AUDIT factually correct?
(2) Is it complete enough for a thesis-standard referee review?
(3) Did it miss any material implementation / identification / robustness / academic-integrity issues?

This is an audit OF the audit, not just a rerun of the audit.

========================================================
1) Non-negotiable rules
========================================================
(1) Fresh-context discipline
- Do not inherit assumptions from the first audit.
- Do not assume its summaries, labels, issue severity, or conclusions are correct.
- Use the first audit as an object to test, not a source of truth.

(2) Hard verification only
Every factual statement in your output must be supported by:
    (a) a code reference (file path + function + relevant lines if easy), OR
    (b) a reproducible command you executed and logged, OR
    (c) a directly inspected artifact/output with path and relevant content summarized.
If you cannot verify something, mark it UNVERIFIED and explain exactly why.

(3) Red-team posture
You are adversarial toward BOTH:
- the implementation, and
- the first-layer audit.

You must actively search for:
- things the first audit failed to notice,
- things it stated too confidently,
- things it labeled incorrectly,
- issues it underweighted,
- issues it overweighted,
- places where it mistook naming/comments for implementation,
- places where it confused theoretical best practice with what code actually does,
- places where it documented code paths that are not actually executed by this suite.

(4) Separate facts from judgments
You must explicitly distinguish:
- VERIFIED FACT
- VERIFIED ERROR IN FIRST AUDIT
- VERIFIED UNDERLYING IMPLEMENTATION ISSUE
- VERIFIED MISSED ISSUE
- VERIFIED FALSE POSITIVE IN FIRST AUDIT
- REFEREE JUDGMENT
- UNVERIFIED CONCERN

(5) No rubber-stamping
Do not praise the first audit unless you have actually tested and confirmed the relevant part.
Do not accept a section as complete merely because it exists.
Do not confuse verbosity with rigor.

(6) No fabricated completeness
If you did not independently test a claim from the first audit, do not say it was confirmed.
If the first audit makes a reasonable claim but lacks evidence, you must flag the evidence gap even if the claim may be true.

(7) Deterministic output
Produce ONE Markdown file at:
docs/provenance/{{SUITE_ID}}_red_team.md

Use stable headings + tables. Avoid prose dumping.

========================================================
2) Primary red-team objectives
========================================================
You must evaluate the first-layer audit on all of the following dimensions:

A. Factual correctness
- Are its statements about model type, variables, filters, FE, SEs, timing, merges, outputs, and reproducibility actually correct?

B. Evidence quality
- Does each material claim rest on actual code/commands/artifacts?
- Does it cite enough to be audit-safe?
- Did it verify counts, match rates, duplicates, missingness, and warnings, or merely describe likely behavior?

C. Completeness
- Did it cover every estimation variable?
- Did it cover every merge/sample-construction step?
- Did it cover model-family-specific validity threats?
- Did it fully assess identification, robustness, and inference?

D. Severity calibration
- Were the most serious problems actually treated as serious?
- Were minor documentation gaps overstated relative to identification or inference threats?
- Did it fail to escalate any thesis-killing issue?

E. False positives / unfair criticism
- Did the first audit accuse the code of problems that are not actually present?
- Did it state concerns as facts when they were only possibilities?
- Did it misread an implementation detail?

F. False negatives / omissions
- What important issues did the first audit miss entirely?
- What hidden dependencies, silent sample changes, identification problems, inference errors, or reproducibility hazards were not flagged?

G. Internal consistency
- Does the first audit contradict itself across sections?
- Do its issue register, verdict, and evidence line up?
- Do its variable definitions match its model descriptions?
- Are the severity labels consistent with the final readiness conclusion?

H. Thesis-standard sufficiency
- If a committee member or journal referee relied only on the first audit, would they be properly informed about the true credibility of this suite?
- Or would they be misled by omissions, understatements, overstatements, or missing caveats?

4) Required working method (do this in order)
========================================================

Step 1 — Open and decompose the first-layer audit
- Read {{FIRST_LAYER_AUDIT_DOC_PATH}} carefully.
- Extract all material claims, judgments, issue labels, severity levels, and final verdicts.
- Build an internal list of claims to verify.
- Identify where the first audit appears:
  * too certain,
  * too vague,
  * incomplete,
  * or suspiciously light/heavy.

Step 2 — Independently identify the suite boundary from source
- Open {{SUITE_TEST_SCRIPT_PATH}}.
- Determine:
  * what dataset/panel/risk-set it loads,
  * which variables it uses,
  * which models/specs it runs,
  * which outputs it writes.
- Do this independently before trusting the first audit’s framing.
- Compare your independent suite boundary against the first audit’s boundary.

Step 3 — Re-trace upstream dependencies
- For each variable in each spec, find its construction and upstream raw fields.
- Independently verify merges, sample filters, reshaping, event-time/risk-set/instrument construction, estimator wrappers, and output builders.
- Note every place where the first audit:
  * omitted a step,
  * compressed a step too much,
  * misstated the step,
  * or failed to verify the step.

Step 4 — Re-run targeted verification
Do not blindly replicate the entire first audit; target the most fragile and high-stakes areas first.

You MUST test, where relevant:
- row counts by stage
- uniqueness of key identifiers before merges
- duplicate contamination risks
- merge match rates
- missingness for DV/key RHS/filters
- FE-group variation support
- cluster counts / cluster-size sanity
- sample losses from lags, winsorization, event windows, or censoring
- timing sanity for leads/lags/outcomes
- warnings, convergence messages, dropped observations, singularity messages
- whether documented outputs correspond to actual executed code paths
- whether caches/stale artifacts/manual steps affect reproduction

Step 5 — Attack the first audit’s judgments
For each material conclusion in the first audit, ask:
- Is this actually proven?
- Is the evidence enough?
- Is the claim too strong?
- Is severity too low or too high?
- Is something more important missing?
- Would a hostile referee agree with this prioritization?

Step 6 — Write the red-team report
Populate sections A–P incrementally.
Do not wait until the end to write.
Your final document must be readable as:
- an audit of the first audit, and
- an independent adversarial referee memo.

========================================================
5) Red-team attack checklist (you must explicitly test these)
========================================================

You must explicitly search for failures in the first audit along these dimensions:

(1) Claim-evidence mismatch
- claim made without direct evidence
- evidence insufficient for severity/conclusion
- claim supported only by naming/comments, not execution

(2) Boundary errors
- wrong entrypoint
- wrong panel artifact
- undocumented side scripts
- code paths documented but not actually used

(3) Sample accounting failures
- undocumented row losses
- silent drops
- mismatch between estimation sample and documented sample
- wrong unit of observation
- hidden subgroup exclusions
- FE-induced sample changes not documented

(4) Merge/provenance failures
- duplicate keys not actually checked
- many-to-many risk not substantiated
- match rates asserted but not verified
- source fields not actually traced to raw inputs

(5) Variable-dictionary failures
- missing spec variables
- incomplete formulas
- wrong source fields
- missing timing alignment
- variable labels inconsistent with code

(6) Inference failures
- clustering issues underweighted
- too few clusters missed
- mismatch between treatment variation and cluster level
- heteroskedasticity/autocorrelation handling misdescribed
- warnings or estimation failures omitted

(7) Identification failures
- post-treatment controls missed
- future leakage missed
- reverse-causality exposure underweighted
- bad sample conditioning missed
- event-time or treatment timing errors missed
- weak-IV / pretrend / bandwidth concerns not demanded when relevant

(8) Robustness failures
- absence of robustness insufficiently criticized
- placebo/falsification missing but not flagged
- alt-cluster / alt-FE / alt-window dependence underweighted
- multiple-testing / spec-search risk ignored

(9) Reproducibility failures
- commands incomplete or untested
- outputs not fully enumerated
- stale cached artifacts not considered
- environment assumptions omitted
- local-path/manual dependencies missed

(10) Audit-craft failures in the first audit
- internal contradictions
- duplicated issues split across sections
- major issue buried as minor note
- final verdict not justified by issue register
- too much prose, too little proof
- weak distinction between fact and judgment
