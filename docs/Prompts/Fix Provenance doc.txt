Claude Code Prompt: Third-Layer Provenance Document Overhaul (Evidence-Driven, Code-Verified)

You are a meticulous technical writer, econometrics documentation specialist, and thesis-grade provenance engineer. You are operating in a FRESH CONTEXT. Your task is to produce a DEFINITIVE, THESIS-STANDARD provenance document for one estimation suite by:

(1) Reading and internalising the RED-TEAM AUDIT findings for this suite,
(2) Independently verifying every claim against the CURRENT codebase (not trusting the red-team audit blindly either),
(3) Rewriting the provenance document so that every section is factually correct, fully evidenced, internally consistent, and sufficient for a hostile thesis referee.

You are NOT patching the old document. You are producing a COMPLETE REPLACEMENT that incorporates all verified red-team findings, discards stale content, and fills every gap the red-team identified.

========================================================
0) Mission
========================================================
Produce a provenance document that:

(1) A hostile thesis referee could rely on as a standalone description of this suite's data, methods, implementation, and limitations.
(2) Passes every check the red-team audit demanded.
(3) Contains ZERO claims that are not independently verified against the current code on disk.
(4) Makes the suite's strengths AND weaknesses transparent — no spin, no omissions, no overstatement.
(5) Is internally consistent across all sections — no contradictory counts, no mixed-run data, no stale references.

========================================================
1) Non-negotiable rules
========================================================

(1) Code-is-truth discipline
- Every factual claim (variable formula, sample filter, merge key, FE structure, SE type, model count, column count, output path) must be verified against the CURRENT source code on disk.
- If the red-team audit says the code does X, verify it yourself before writing it into the provenance doc.
- If the current provenance doc says Y but the code says Z, write Z and note the correction.
- NEVER copy-paste claims from the old provenance doc without re-verifying them.

(2) Red-team findings are inputs, not commands
- Read the red-team audit carefully. It identifies errors, omissions, and gaps.
- For each finding: verify it independently, then either (a) fix the provenance doc accordingly, or (b) explain in the doc why the finding does not apply.
- Do NOT blindly accept red-team claims. Some may be wrong or overstated. You must verify.
- Do NOT ignore red-team claims. Every material finding must be addressed — either fixed or explicitly rebutted with evidence.

(3) Completeness standard
- Every section listed in the deliverable structure MUST be present and substantive.
- Empty sections, placeholder text, "TBD", "pending re-run", or "see above" are FORBIDDEN.
- If information is genuinely unavailable (e.g., no fresh run exists), state this explicitly as a known gap with a concrete remediation path.

(4) Internal consistency
- Sample sizes, model counts, variable lists, column counts, and output paths must be consistent across ALL sections.
- If Section A says 8 models and Section F says 16, that is a failure.
- Cross-reference every quantitative claim across sections before finalising.

(5) Precision over prose
- Prefer tables over paragraphs.
- Prefer exact code references (file:function:line) over vague descriptions.
- Prefer verified numbers over qualitative statements.
- Every variable formula must show the actual computation, not a textbook definition.

(6) Honest about limitations
- The provenance doc must have a frank, prominent section on known limitations and unresolved issues.
- Do not bury limitations in footnotes or appendices.
- A referee who reads only the limitations section should understand the suite's main vulnerabilities.

(7) No cosmetic padding
- Do not add sections, subsections, or tables that contain no real information.
- Do not inflate the document with boilerplate, generic econometric warnings, or textbook recitations.
- Every sentence must earn its place by conveying suite-specific, verified information.

========================================================
2) Required working method (do this in order)
========================================================

Step 1 — Read the red-team audit
- Open {{RED_TEAM_AUDIT_PATH}}.
- Extract every finding: verified errors, missed issues, false positives, severity recalibrations, completeness gaps, reproducibility concerns, and priority fixes.
- Build an internal checklist of everything the new provenance doc must address.

Step 2 — Read the current provenance doc
- Open {{CURRENT_PROVENANCE_DOC_PATH}}.
- Identify what is reusable (after re-verification) vs. what must be rewritten or discarded.
- Note any sections that are missing entirely.

Step 3 — Independently verify the suite boundary
- Open {{SUITE_TEST_SCRIPT_PATH}}.
- Determine: what panel it loads, what variables it uses, what models/specs it runs, what outputs it writes, what FE/SE/filters it applies.
- Open {{PANEL_BUILDER_PATH}}.
- Determine: what raw data sources it reads, what merges it performs, what variables it constructs, what filters it applies, what the output schema is.
- Trace key variables through shared utilities (e.g., _compustat_engine.py, financial_utils.py, variables/*.py).

Step 4 — Cross-check red-team findings against code
- For each material red-team finding, verify it against the current code.
- Mark each finding as: CONFIRMED (fix in doc), PARTIALLY CONFIRMED (fix with nuance), NOT CONFIRMED (explain why), or NO LONGER APPLICABLE (code has changed).

Step 5 — Write the new provenance document
- Write all sections in order, populating each with verified information.
- After writing, do a consistency pass: check that all cross-references (counts, variable lists, model numbers, output paths) agree across sections.

Step 6 — Final quality gate
- Re-read the red-team audit's "Priority Fixes" and "What a Committee Would Not Know" sections.
- Verify that EVERY item is now addressed in the new document.
- If any item cannot be addressed (e.g., requires a fresh pipeline run), document it explicitly in the Known Limitations section.

--- SECTION 11: Identification & Inference Assessment ---
This section must be written from a thesis-referee perspective.
Address ALL of the following (mark N/A if genuinely not applicable, with explanation):

- Identification strategy: What variation identifies the effect? Within-firm? Within-industry? Cross-sectional?
- Endogeneity concerns: Is the key IV plausibly exogenous? What are the main threats?
- Reverse causality: Could the DV cause the IV?
- Post-treatment controls: Are any controls potentially affected by the treatment?
- Timing alignment: Are IVs measured before DVs? Any look-ahead risk?
- Moulton problem: Is the DV constant within clusters while IVs vary at a finer level (or vice versa)?
- Clustering adequacy: Is the cluster level appropriate? Are there enough clusters?
- Multiple testing: How many tests does this suite run? What is the expected false-positive count under the null?
- Robustness: What robustness checks exist? What is missing?
- Interpretation discipline: Are findings described as predictive associations or causal effects?

For each concern, state: (a) the concern, (b) how the implementation addresses it (or doesn't), (c) the residual risk, (d) what a referee should know.

--- SECTION 14: Design Decisions & Rationale ---

Each decision: what was chosen, why, what alternative was considered, what the tradeoff is.
