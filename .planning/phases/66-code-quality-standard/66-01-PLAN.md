Add Section 1: Naming Conventions to CODE_QUALITY_STANDARD.md with:

### 1.1 Script Naming Convention (NAM-01)
- Pattern: `{Stage}.{Step}_{Description}.py`
- Stage: Single digit (1-4)
- Step: Single decimal (0, 1, 2, etc.)
- Description: PascalCase (e.g., BuildSampleManifest, H1Variables)
- Examples from codebase:
  - `1.0_BuildSampleManifest.py`
  - `3.1_H1Variables.py`
  - `4.11_H9_Regression.py`
- Rationale: Natural sort order, immediate context

### 1.2 Module Naming Convention (NAM-02)
- Pattern: snake_case for module files
- Package directories: snake_case (shared/, observability/)
- Version variants: lowercase v + number (v1/, v2/)
- Anti-patterns to avoid:
  - camelCase modules: panelOls.py
  - Abbreviated names: pu.py, ut.py
  - Mixed conventions: Panel_OLS.py

### 1.3 Function/Class Naming Convention (NAM-03)
- Functions: snake_case (get_latest_output_dir, run_panel_ols)
- Classes: PascalCase (CollinearityError, DualWriter)
- Constants: UPPER_SNAKE_CASE (MAX_RETRIES, DEFAULT_TIMEOUT)
- Private functions: _leading_underscore (_check_thin_cells)
- Source reference: PEP 8 Naming Conventions

### 1.4 Variable Naming Patterns (NAM-04)
- DataFrame: descriptive_df or just descriptive (df, manifest_df, panel_df)
- Series: descriptive_s or just descriptive (returns_s, uncertainty)
- Counts: n_ prefix or _count suffix (n_firms, record_count)
- Indices: i_, j_, k_ for simple loops (i_row, j_col)
- Boolean: is_, has_, should_ prefixes (is_valid, has_missing)
- Paths: _path or _dir suffix (output_path, input_dir)

### 1.5 Output File Naming Patterns (NAM-05)
- Timestamp: ISO 8601 compact (20260213_143052)
- Date-only: ISO 8601 (2026-02-13)
- Checksum: SHA-256 (12 chars) (a3f2b8c91d2f)
- Version: v + number (v1, v2)
- Examples:
  - `4_Outputs/1.0_BuildSampleManifest/2026-02-13-143052/master_manifest.parquet`
  - `data/processed/manifest_20260213_a3f2b8c91d2f.parquet`

Include rationale subsection explaining why these conventions were chosen.

### 5.3 No Bare Except (Critical)
Reference PEP 760 (2024) - No More Bare Excepts

Anti-pattern (NEVER use):
```python
try:
    operation()
except:  # WRONG - catches SystemExit, KeyboardInterrupt
    pass
```

Correct patterns:
```python
# Specific exception
try:
    operation()
except ValueError as e:
    logger.error(f"Invalid value: {e}")
    raise

# Re-raise with context
try:
    risky_operation()
except FileNotFoundError as e:
    raise DataValidationError(f"Required file not found: {e}") from e
```

### 5.4 Key Rules
1. Never use bare `except:` (PEP 760)
2. Always specify exception type(s)
3. Either handle or re-raise
4. Use `raise ... from e` for exception chaining
5. Log before re-raising when appropriate

### 6.1 Function Size Guidelines (CODE-05)

| Guideline | Recommendation | Rationale |
|-----------|----------------|-----------|
| Target length | 20-30 lines | Readability, testability |
| Maximum length | 50 lines | Hard limit requiring justification |
| Single Responsibility | One thing well | More important than line count |
| Cognitive complexity | < 10 | Avoid nested conditionals |

### 6.2 Warning Signs
- Function requires scrolling to read
- Multiple levels of indentation (>3)
- More than 3-4 parameters
- "And" in function name (`load_and_validate_and_transform`)

### 6.3 Module Organization Rules
- One module = one concern
- Maximum ~500 lines per module (soft limit)
- Group related functions together
- Private helpers at bottom
- Public API at top or in __init__.py

### 6.4 When to Split
Split a function when:
- It does more than one distinct thing
- It's hard to name concisely
- It requires extensive comments to understand
- Testing requires mocking multiple dependencies

### 6.5 Module Organization Pattern
```python
# 1. Module docstring
# 2. Imports (organized per Section 4)
# 3. Module-level constants
# 4. Public classes
# 5. Public functions
# 6. Private helper functions ( _leading_underscore )
```

### 6.6 Rationale
- Readability: Functions should fit on one screen
- Testability: Small functions are easier to test
- Maintainability: Clear separation of concerns
- Code review: Easier to review small, focused changes
