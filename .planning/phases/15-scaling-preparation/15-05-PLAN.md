    ## Overview
    Document the current capacity of the F1D data pipeline and paths for scaling to larger datasets.

    ## Current Limits (As of Phase 15)

    ### Dataset Size Limits
    - **Transcripts**: ~50K transcripts (2002-2018)
    - **Rows per transcript**: ~500 average rows
    - **Total rows**: ~25M rows across all steps
    - **Parquet file sizes**: 100MB - 1GB per year

    ### Memory Requirements
    - **Minimum**: 8GB RAM (small steps)
    - **Recommended**: 16GB RAM (full pipeline)
    - **Peak memory**: ~4GB during entity linking (1.2)

    ### Processing Time
    - **Full pipeline**: ~2-4 hours on modern hardware
    - **Per step**: 5-30 minutes
    - **Parallelization**: Single-threaded (thread_count=1 in config)

    ## Scaling Improvements (Phase 15)

    ### 1. Deterministic Parallelization (15-01)
    - **What Added**: parallel_utils.py with SeedSequence spawning
    - **Benefit**: Parallel workers maintain reproducibility
    - **When to Use**: Add ProcessPoolExecutor to CPU-bound operations
    - **Example**: Tokenization, fuzzy matching, bootstrapping

    ### 2. Column Pruning (15-02)
    - **What Added**: Column-specific Parquet reads in 1.2, 1.4, 3.2
    - **Benefit**: 30-50% memory reduction for large files
    - **When to Use**: When scripts need subset of columns
    - **Example**: Entity linking only reads name columns, not full transcript

    ### 3. Memory-Aware Throttling (15-03)
    - **What Added**: MemoryAwareThrottler class in chunked_reader.py
    - **Benefit**: Automatic chunk size adjustment based on memory pressure
    - **Configuration**: config/project.yaml > chunk_processing section
    - **When to Use**: Processing files >10GB on <32GB RAM systems

    ### 4. Enhanced Memory Tracking (15-04)
    - **What Added**: track_memory_usage decorator, memory_mb in stats.json
    - **Benefit**: Identify memory-intensive operations
    - **When to Use**: All scripts processing large datasets

    ## Scaling Paths by Use Case

    ### Path A: 2x Dataset Size (~100K transcripts)
    **Current**: 50K transcripts, 16GB RAM sufficient

    **Improvements**:
    1. Enable parallel processing: set `enable_parallel: true` in config
    2. Increase chunk size to 50K rows (config > chunk_processing > base_chunk_size)
    3. Add column pruning to remaining scripts (2.2, 2.3, 3.x)
    4. Monitor memory with memory_mb stats (Phase 15-04)

    **Expected**: 16GB RAM still sufficient, ~1.5x processing time

    ### Path B: 10x Dataset Size (~500K transcripts)
    **Current**: 50K transcripts, 16GB recommended

    **Improvements**:
    1. Enable parallel processing with 4-8 workers
    2. Set max_memory_percent to 60% (aggressive throttling)
    3. Reduce base_chunk_size to 5K rows (smaller chunks)
    4. Add column pruning to all scripts reading Parquet files
    5. Use PyArrow Dataset API for lazy loading (read_dataset_lazy)
    6. Consider SSD storage for faster I/O

    **Expected**: 32GB RAM required, ~3x processing time

    ### Path C: 100x Dataset Size (~5M transcripts)
    **Current**: Not feasible (would exceed memory)

    **Major Changes Required**:
    1. Switch to distributed processing (Dask or Ray)
    2. Incremental database storage (PostgreSQL/ClickHouse) instead of Parquet
    3. Stream processing architecture (Apache Flink or Spark Streaming)
    4. Revisit pipeline design for streaming vs batch

    **Note**: This requires significant architectural changes beyond Phase 15 scope.

    ## Configuration Recommendations

    ### Small System (8GB RAM)
    ```yaml
    determinism:
      thread_count: 1
    chunk_processing:
      max_memory_percent: 70.0
      base_chunk_size: 5000
      enable_throttling: true
    ```

    ### Medium System (16GB RAM - Current Standard)
    ```yaml
    determinism:
      thread_count: 1
    chunk_processing:
      max_memory_percent: 80.0
      base_chunk_size: 10000
      enable_throttling: true
    ```

    ### Large System (32GB RAM - 2x-5x datasets)
    ```yaml
    determinism:
      thread_count: 4
    chunk_processing:
      max_memory_percent: 85.0
      base_chunk_size: 50000
      enable_throttling: true
    ```

    ## Bottleneck Analysis

    ### Current Bottlenecks (Identified in Phase 15)
    1. **Entity Linking (1.2)**: Fuzzy matching is O(n²) - most CPU-intensive
    2. **Tokenization (2.1)**: Text processing is memory-intensive for long transcripts
    3. **Market Variables (3.2)**: CRSP data loads are I/O intensive
    4. **Merging Operations**: Large intermediate results (1.4, 2.2)

    ### Improvement Priority
    1. **High**: Add parallelization to fuzzy matching (1.2)
    2. **Medium**: Add chunking to tokenization (2.1)
    3. **Medium**: Add column pruning to remaining scripts
    4. **Low**: Optimize merge operations (requires data structure changes)

    ## Monitoring and Debugging

    ### Memory Monitoring
    - Check stats.json > memory_mb section for operation-level memory
    - Use MemoryAwareThrottler.log_memory_status() for real-time tracking
    - Set log_memory_status: true in config for verbose logging

    ### Performance Monitoring
    - Check stats.json > processing_duration for total time
    - Use track_memory_usage decorator to identify slow operations
    - Benchmark with different chunk sizes (1K, 5K, 10K, 50K rows)

    ### Troubleshooting OOM Errors
    1. Reduce max_memory_percent to 60-70%
    2. Reduce base_chunk_size by 50%
    3. Add column pruning to reduce memory per chunk
    4. Disable parallel processing (thread_count: 1)
    5. Upgrade RAM or use machine with more memory

    ## References

    - NumPy Parallel RNG: https://numpy.org/doc/stable/reference/random/parallel.html
    - PyArrow Dataset API: https://arrow.apache.org/docs/python/dataset.html
    - Phase 15-RESEARCH.md: Scaling research and patterns
    - config/project.yaml: Configuration parameters

    Include date created and reference to Phase 15 completion.
